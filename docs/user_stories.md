# SQL Agent — 使用者故事

## 角色定義

- **SA**：系統分析師，需要快速產出完整的資料庫規格文件
- **PM**：專案經理，需要確認設計符合業務需求，不需手寫 SQL
- **Dev**：後端開發者，需要可直接使用的 DDL 與欄位定義
- **Admin**：系統管理員，負責帳號管理與系統設定

---

## 設計模式（Design Mode）

### US-01：快速建立資料表規格書

**身份**：SA  
**需求**：我想描述一個新資料表的需求，讓 AI 幫我整理成完整的規格書  
**價值**：不需手動整理欄位清單，節省 30 分鐘以上的文件作業時間

**驗收標準**：
- 使用者輸入一段自然語言需求後，AI 在 5 輪內追問完所有關鍵欄位細節
- AI 最終產出規格書包含：欄位名稱、型態、長度、NULL、預設值、PK/FK/UNIQUE/INDEX 旗標、說明
- 產出文件可直接下載為 `.md`

---

### US-02：確認 AI 理解是否正確

**身份**：PM  
**需求**：我想在產出文件前，確認 AI 整理的 Schema 是否符合我的業務描述  
**價值**：避免產出錯誤文件，需要時可直接返回修改

**驗收標準**：
- 確認頁左欄顯示 AI 整合的需求摘要（非原始對話）
- 確認頁右欄以表格格式顯示所有資料表與欄位
- 使用者可點「返回補充需求」，對話記錄不遺失

---

### US-03：取得可直接執行的 DDL

**身份**：Dev  
**需求**：我想直接拿到 PostgreSQL DDL 腳本，不需要自己從規格書轉寫  
**價值**：節省手動撰寫 CREATE TABLE 的時間，避免語法錯誤

**驗收標準**：
- DDL 包含 `CREATE TABLE`、`INDEX`、`Migration`（`IF NOT EXISTS`）、`COMMENT ON COLUMN`
- DDL 在 PostgreSQL 14+ 可直接執行，無語法錯誤
- DDL 頁籤支援語法高亮與一鍵複製

---

### US-04：設計參考現有 DB 結構

**身份**：Dev  
**需求**：我想在設計新資料表時，讓 AI 知道現有的 DB 結構，避免欄位或命名衝突  
**價值**：設計與現有系統一致，減少後期對接問題

**驗收標準**：
- 建立 session 時可填入 PostgreSQL 連線字串，自動匯入現有 DB 結構
- AI 在對話中參考現有 DB 結構提供建議（例如沿用相同主鍵命名規範）
- 確認頁顯示設計 Schema 與現有 DB 的差異（新增/刪除/變更欄位）

---

### US-05：版本回溯

**身份**：SA  
**需求**：我想在多次修改後，回到之前某個版本的 Schema 設計  
**價值**：修改過程中不怕覆蓋，可自由探索不同設計方向

**驗收標準**：
- 每次 AI 產出新的 Schema 自動建立版本快照
- 確認頁顯示所有歷史版本（含資料表數量），可一鍵還原

---

## 審查模式（Review Mode）

### US-06：AI 分析現有 DB 設計問題

**身份**：SA  
**需求**：我想讓 AI 分析現有資料庫的設計，找出潛在問題  
**價值**：快速取得第三方視角的設計建議，不需請 DBA 人工 Review

**驗收標準**：
- 使用「審查模式」建立 session，填入 PostgreSQL 連線字串
- AI 自動分析並在 60 秒內產出審查報告
- 報告含設計一致性、資料完整性、效能考量、安全性四個維度
- 每條建議標明資料表與欄位名稱，說明問題與改善方向
- 報告含整體評分（X/10）

---

## Session 管理

### US-07：查閱歷史專案

**身份**：PM  
**需求**：我想在首頁查看所有過去的設計專案，快速找到需要的文件  
**價值**：無需記憶每個 URL，集中管理所有設計工作

**驗收標準**：
- 首頁列出所有 session，顯示標題、建立日期、狀態、資料表數
- 支援篩選：全部 / 進行中 / 待確認 / 已完成
- 可直接點擊卡片進入 session
